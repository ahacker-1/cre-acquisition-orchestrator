# Real-World Pile — adversarial ingestion fixtures

A deliberately nasty "pile" of files a real CRE operator drops into a deal
folder: a messy-but-valid core deal plus the junk that always comes with it
(zips, photos, brochures, mislabeled blobs, corrupt files, empties, huge files).

Generated by `scripts/build-real-world-pile.py` (run with a Python that has
`openpyxl` + `reportlab`). Some files are copies of existing fixtures under
`fixtures/parsers/`; the rest are synthesized junk. The oversized
`huge-rent-roll.csv` is **not committed** (a ~35 MB file does not belong in git);
it is regenerated at test time by `ensureOversizedCsv` and is gitignored.

The contract under test (`scripts/real-world-pile.test.mjs`, run via
`npm run test:pile`): **every file must degrade gracefully.** Acceptable terminal
states are `extracted`, `extraction-pending`, `unsupported`, `parse_failed`, or
`parser-unavailable`. A file must **never** crash past the parser boundary, hang
(native CSV is guarded by a 25 MB size cap; Python parsers by a 15s timeout),
silently return `extracted` with zero fields, leak a local filesystem path in an
error, or return high confidence on a garbage value.

Classification (`workspace-service.ts::classifyDocument`) is **content-aware** for
the two tabular financial types: a clear rent-roll / T12 content signal overrides
an ambiguous or wrong filename guess (so a mislabeled file is routed by what it
*is*, not just what it is named). Other types fall back to filename routing.

## Environment note
The local Python parsers (`parse_excel.py`, `parse_pdf.py`) need `openpyxl` /
`pdfplumber`. `parser-service.ts` tries interpreters in order and is resilient:
an interpreter that is missing or lacks the dependencies is skipped in favor of
the next one, and only if *no* available interpreter has the deps does it report
a clear `parser-unavailable` (with local paths redacted) rather than a crash.
